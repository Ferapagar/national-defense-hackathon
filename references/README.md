# References

This folder collects reference implementations, datasets, and working notes that support the project. The main item currently present is a homography and image-warping demo under `references/homographies/code`.

### `references/homographies/code`

This directory contains a compact Python pipeline for estimating a homography from hand-labelled point correspondences, warping one image into another image plane, and blending the results into a simple mosaic.

The folder is organized as follows:

- `main.py` contains the full runnable demo.
- `test.py` is a minimal scratch file unrelated to the homography pipeline.
- `README.txt` is the original short note for the demo; this document expands that into a proper guide.
- `correspondances/` stores the point-pair annotations used to compute each transformation.
- `photos/` stores the source images.
- `result_mosaics/` stores example outputs generated by the pipeline.

## What This Reference Implements

The homography demo is a classical two-image stitching pipeline.

At a high level it does the following:

1. Load two images from `photos/`.
2. Manually load the matching pixel coordinates for those images from `correspondances/`.
3. Solve for the homography matrix `H` that maps the first image into the second image coordinate frame.
4. Warp the first image using inverse mapping and bilinear interpolation.
5. Combine the warped image with the second image to form a mosaic.
6. Save intermediate and final images to disk.

This makes the folder a useful reference for any workflow that needs planar alignment, perspective correction, or a lightweight image-stitching baseline. The two images do not need to be only translated versions of each other; a homography can also handle in-plane rotation, translation, scale, shear, and general perspective effects as long as the scene is approximately planar.

## File-by-File Guide

### `main.py`

This is the entry point for the demo. It defines the functions used by the pipeline and then runs a concrete example with the elevator images.

Key functions:

- `compute_H(pts1, pts2)` builds the linear system for the homography and solves it with least squares.
- `bilinear_interpolation(image, coords)` samples the source image at continuous coordinates.
- `get_image_corners(im, H)` projects the four corners of an image through a homography to estimate the output bounding box.
- `warp_image(image, H)` creates the warped image canvas and fills it using inverse warping.
- `naive_join(images, offsets_y, offsets_x)` overlays two images with a simple zero-fill policy.
- `get_full_size_separate_images(images, masks, offsets_y, offsets_x)` expands each image into a shared canvas so the final blend can be computed consistently.

The script then executes a full example using the elevator image pair.

### `correspondances/*.json`

Each JSON file stores manually established point correspondences for a specific image pair.

Typical structure:

```json
{
  "im1_name": "window",
  "im2_name": "great_hall_1",
  "im1Points": [[66, 116], [409, 6]],
  "im2Points": [[156, 75], [123, 96]]
}
```

Fields:

- `im1_name` identifies the first source image without the extension.
- `im2_name` identifies the second source image without the extension.
- `im1Points` contains the pixel coordinates selected in the first image.
- `im2Points` contains the matching pixel coordinates selected in the second image.

The two point lists must contain the same number of correspondences and should be ordered so that each pair refers to the same physical feature.

## How The Homography Is Computed

The homography `H` is a $3 \times 3$ projective transformation matrix that maps homogeneous coordinates from one plane to another.

In practice, that means the related images can differ by rotation and translation, and more generally by any planar projective warp.

For each correspondence pair `(x, y) -> (x', y')`, the code adds two linear equations to the system.
With `n` correspondences, the solver builds a `2n x 8` matrix and solves for the eight free parameters of the homography, fixing the last entry to `1` to remove scale ambiguity.

This is the standard direct linear transform style setup for a planar transformation.

## Image Warping Strategy

The warping implementation uses inverse mapping rather than forward splatting.

Why this matters:

- Every output pixel is assigned a value.
- Holes are avoided because the code samples from the source image by tracing backward through `H^{-1}`.
- Bilinear interpolation is used to reduce aliasing and make the output smoother.

The code then estimates a bounding rectangle for the warped image and uses that rectangle to create a canvas large enough to hold the transformed result.

## Blending Strategy

The current mosaic blend is intentionally simple.

- Warped images are placed on a common canvas.
- Pixels are accumulated where masks are valid.
- The final output is normalized by the sum of the masks.

This is enough for a reference implementation and small controlled examples, but it does not attempt advanced seam finding, exposure compensation, or multi-band blending.

## Run Instructions

Run the demo from `references/homographies/code` so that the relative paths resolve correctly.

```bash
python main.py
```

The script currently uses the elevator correspondence pair by default:

- `photos/elevator_1.png`
- `photos/elevator_2.png`
- `correspondances/elevator_1_elevator_2.json`

If you change the file names in the script, make sure the image paths and correspondence JSON stay aligned.

## Outputs

Running `main.py` writes the following files into the working directory:

- `im1warp.jpg` — warped version of the first image.
- `im12warped.jpg` — naive overlay of the warped image and the second image.
- `im1warpedlarge.jpg` — first warped image expanded to the common canvas.
- `im2large.jpg` — second image expanded to the common canvas.
- `finalmosaic.jpg` — blended result using mask-weighted averaging.

## Customization Notes

The script contains a few hardcoded values that are meant to be edited when trying a different pair of images:

- `im1_name` and `im2_name` select the source images.
- The correspondence file in `with open(...)` selects the matching annotation set.
- The commented-out code near the bottom shows how to switch to the `window` example.

When adding a new image pair, the minimum required steps are:

1. Add the images to `photos/`.
2. Add a new correspondence JSON to `correspondances/`.
3. Update the filenames in `main.py`.
4. Run the script and inspect the saved outputs in the working directory or `result_mosaics/`.

## Practical Limitations

This reference is intentionally small and experimental, so there are some limitations to keep in mind:

- It assumes the images are related by a planar projective transform or a sufficiently good approximation.
- It depends on manual correspondence selection, so quality depends on annotation accuracy.
- It does not perform feature detection or automatic match extraction.
- It does not include robust outlier rejection such as RANSAC.
- It does not include advanced blending, so visible seams can remain.

## Suggested Use Cases

Use this reference when you need:

- a lightweight example of homography estimation,
- a baseline for perspective warping,
- a starting point for panorama stitching,
- a reproducible manual correspondence workflow,
- or a compact example for teaching projective geometry.

If you want to extend this folder into a more complete stitching tool, the next natural improvements are automatic feature matching, RANSAC-based outlier filtering, and seam-aware blending.
